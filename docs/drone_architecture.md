# 3.2 Архитектура программного комплекса управления информационной безопасностью кибер-физической системы на базе целевой платформы Holybro X500 v2 + Pixhawk 6C + PX4

## 3.2.1 Назначение и место программного комплекса в составе системы

Для исследования информационной безопасности кибер-физической системы в проекте реализован локальный стенд класса `SITL`, в котором аппаратные компоненты заменены программными аналогами, а функции мониторинга и анализа вынесены во внешний программный контур. В конфигурации стенда целевая аппаратная платформа задана как `Holybro X500 v2` и `Pixhawk 6C`, однако аппаратная валидация этой платформы в текущем репозитории не подтверждена.  
Подтверждение по исходному коду: [sim_stack.json](../sim_stack.json), строки 2-13; [architecture_report.py](../tools/architecture_report.py), строки 119-175; [hardware_validation.example.json](../validation/hardware_validation.example.json), строки 1-25.

В реализованной конфигурации стенда роль летательного аппарата представлена моделью `gz_x500`, а роль полетного контроллера выполняет процесс `PX4 SITL`. В качестве внешнего программного комплекса информационной безопасности используется `SecurityMonitorApp`, который получает данные через `MAVSDK` и при включенном шлюзовом режиме подключается к PX4 через промежуточный `MavlinkGateway`. Обмен в стенде организован по `MAVLink/UDP`.  
Подтверждение по исходному коду: [sim_stack.json](../sim_stack.json), строки 3-25; [launch_stack.py](../launch_stack.py), строки 127-159, 183-195; [monitor.py](../monitor.py), строки 79-102; [security_agent/app.py](../security_agent/app.py), строки 43-57.

В конфигурации стенда также предусмотрено использование `QGroundControl` как наземной станции оператора через ручной `UDP`-link на `127.0.0.1:14552`. По коду и конфигурации подтверждается, что этот компонент включен в архитектуру стенда, однако в приложенных артефактах отсутствует отдельный процессный лог запуска `QGroundControl`, поэтому корректно говорить именно о предусмотренном использовании, а не о подтвержденной аппаратно-полевой эксплуатации.  
Подтверждение по исходному коду: [sim_stack.json](../sim_stack.json), строки 17-22; [launch_stack.py](../launch_stack.py), строки 197-206; [monitor.py](../monitor.py), строки 41-52; [security_agent/app.py](../security_agent/app.py), строки 65-69.

Программный комплекс размещается не внутри полетного контроллера, а как внешний надзорный уровень по отношению к `PX4 SITL`. Он выполняет функции наблюдения, анализа, оценки риска и журналирования, но не заменяет автопилот и не реализует собственные алгоритмы стабилизации и навигации. Утверждение о доказанном сохранении детерминированности PX4 проектом не подтверждается.  
Подтверждение по исходному коду: [security_agent/app.py](../security_agent/app.py), строки 43-109; [tools/architecture_report.py](../tools/architecture_report.py), строки 147-175; [validation/README.md](../validation/README.md), строки 31-43.

## 3.2.2 Архитектура текущей реализации стенда

Структура текущей реализации стенда включает среду моделирования, полетный компонент, операторский контур, протокольный шлюзовой уровень и внешний модуль мониторинга безопасности. В текстовом виде архитектура может быть представлена следующим образом.

**Рисунок 3.2 - Архитектура текущей реализации программного комплекса**

```text
Gazebo / модель gz_x500
        <-> PX4 SITL
             -> GCS channel -> MAVLink Gateway -> QGroundControl (предусмотренный операторский клиент)
             -> API channel -> MAVLink Gateway -> Security Agent / сценарии воздействия
                                           |
                                           -> Telemetry Collector
                                           -> StateGuard
                                           -> Threat Detector
                                           -> Risk Engine
                                           -> Responder
                                           -> Audit Logger
```

Ключевой особенностью текущей реализации является отдельный `MavlinkGateway`, через который разнесены два канала взаимодействия: `GCS` и `API`. Gateway анализирует MAVLink-кадры, формирует протокольные события, ведет статистику канала связи и поддерживает выборочную блокировку сообщений. В текущей конфигурации стенда активирована блокировка `SERIAL_CONTROL`; блокировка записей параметров и миссии реализована в коде, но не включена по умолчанию.  
Подтверждение по исходному коду: [security_agent/gateway.py](../security_agent/gateway.py), строки 82-145, 171-245, 261-321, 323-452; [monitor.py](../monitor.py), строки 83-93; [sim_stack.json](../sim_stack.json), строки 18-25.

Дополнительно в проекте зафиксирована локальная спецификация модели `gz_x500`, подтверждающая соответствие make-цели `gz_x500`, наличие четырех моторных исполнительных каналов и датчиков `air_pressure`, `imu`, `magnetometer` и `navsat`.  
Подтверждение по исходному коду: [validation/gz_x500_model_spec.json](../validation/gz_x500_model_spec.json), строки 2-90; [tools/extract_gz_x500_model_spec.py](../tools/extract_gz_x500_model_spec.py), строки 109-143.

## 3.2.3 Функциональная архитектура программного комплекса

В составе `Security Agent` подтверждаются шесть функциональных модулей: `TelemetryCollector`, `StateGuard`, `ThreatDetector`, `RiskEngine`, `Responder` и `AuditLogger`.  
Подтверждение по исходному коду: [security_agent/app.py](../security_agent/app.py), строки 52-57.

`TelemetryCollector` реализует сбор телеметрии через `MAVSDK` и получает данные о режиме полета, состоянии `armed/disarmed`, координатах, относительной высоте, аккумуляторе, GPS и health-состоянии.  
Подтверждение по исходному коду: [security_agent/collector.py](../security_agent/collector.py), строки 33-164; [logs/telemetry.jsonl](../logs/telemetry.jsonl).

`StateGuard` реализует периодический снимок параметров PX4 и текущей миссии. Для миссии формируется контрольный `SHA-256`-хеш по нормализованному набору точек, что позволяет обнаруживать факт изменения миссии.  
Подтверждение по исходному коду: [security_agent/state_guard.py](../security_agent/state_guard.py), строки 38-145.

`ThreatDetector` анализирует телеметрические, протокольные и канальные события. По коду подтверждаются правила выявления частой смены режима, серий `arm/disarm`, резких скачков координат и высоты, деградации GPS, потери доверия к навигационному состоянию, всплесков командного трафика, попыток изменения параметров, попыток изменения миссии, попыток использования `SERIAL_CONTROL`, а также событий блокировки сообщений шлюзом.  
Подтверждение по исходному коду: [security_agent/detector.py](../security_agent/detector.py), строки 45-389.

`RiskEngine` реализует контекстную оценку риска с учетом фаз `GROUND`, `TAKEOFF_PREP`, `AIRBORNE`, `MISSION`, `RETURN` и `LANDING`.  
Подтверждение по исходному коду: [security_agent/risk_engine.py](../security_agent/risk_engine.py), строки 6-66; [security_agent/models.py](../security_agent/models.py), строки 143-191.

`Responder` реализован частично. В коде предусмотрены рекомендации `log_only`, `alert_operator`, `hold_position`, `block_command_source`, `return_or_land` и `lockdown`, однако фактически исполняемыми через `MAVSDK` являются только `hold_position` и `return_or_land`.  
Подтверждение по исходному коду: [security_agent/responder.py](../security_agent/responder.py), строки 13-35; [sim_stack.json](../sim_stack.json), строки 23-25.

`AuditLogger` записывает телеметрические, протокольные и security/risk-события в отдельные `JSONL`-файлы, что обеспечивает проверяемость работы стенда.  
Подтверждение по исходному коду: [security_agent/audit.py](../security_agent/audit.py), строки 16-43; [logs/telemetry.jsonl](../logs/telemetry.jsonl); [logs/protocol_events.jsonl](../logs/protocol_events.jsonl); [logs/security_events.jsonl](../logs/security_events.jsonl).

Для удобства роли модулей сведены в таблицу 3.4.

## 3.2.4 Информационные потоки, сценарии и логика работы

С точки зрения информационного обмена текущая архитектура строится вокруг потока телеметрии и потока управляющих сообщений. Телеметрия PX4 поступает в клиентские контуры и во внешний модуль мониторинга через `MavlinkGateway`. Команды оператора и внешних программных клиентов также проходят через gateway, что обеспечивает централизованную точку протокольного наблюдения.  
Подтверждение по исходному коду: [security_agent/config.py](../security_agent/config.py), строки 7-29; [security_agent/gateway.py](../security_agent/gateway.py), строки 147-250, 261-321; [monitor.py](../monitor.py), строки 83-99.

Логика работы подтверждается следующей последовательностью:

1. `TelemetryCollector` получает телеметрические события;
2. `StateGuard` периодически снимает параметры PX4 и миссию;
3. `ThreatDetector` анализирует телеметрию, MAVLink-команды и события качества канала;
4. `RiskEngine` вычисляет уровень риска с учетом фазы полета;
5. `Responder` при необходимости выполняет поддерживаемое действие;
6. `AuditLogger` сохраняет события, оценки риска и протокольную историю.

Подтверждение по исходному коду: [security_agent/app.py](../security_agent/app.py), строки 71-109; [security_agent/risk_engine.py](../security_agent/risk_engine.py), строки 6-66; [security_agent/audit.py](../security_agent/audit.py), строки 16-43.

В репозитории подтверждены четыре отдельных сценария воздействия: многократные команды `arm/disarm`, изменение параметров PX4, перезапись миссии и подача агрессивных `offboard`-команд. Кроме того, в проекте присутствуют две безопасные эмуляции для `MAVLink SERIAL_CONTROL`, не отправляющие сообщения в PX4 и предназначенные только для проверки работы детектора и цепочки журналирования.  
Подтверждение по исходному коду: [attack_arm_disarm_flap.py](../scenarios/attack_arm_disarm_flap.py), строки 7-80; [attack_param_tamper.py](../scenarios/attack_param_tamper.py), строки 7-84; [attack_mission_overwrite.py](../scenarios/attack_mission_overwrite.py), строки 7-109; [attack_navigation_excursion.py](../scenarios/attack_navigation_excursion.py), строки 7-82; [simulate_serial_control_detection.py](../scenarios/simulate_serial_control_detection.py), строки 17-89; [simulate_serial_control_log_flow.py](../scenarios/simulate_serial_control_log_flow.py), строки 18-104.

При этом утверждать реализацию сценариев `replay`, `spoofing`, подмены телеметрии, отдельного `DoS`-сценария канала связи и компрометации `GCS` по текущему репозиторию нельзя. Для этих угроз в проекте отсутствуют отдельные сценарии, скрипты или подтверждающие логи.

## 3.2.5 Вывод по архитектуре программного комплекса

Текущая реализация проекта подтверждает внешний надзорный программный комплекс информационной безопасности, встроенный в локальный `SITL`-стенд на базе `PX4 SITL`, модели `gz_x500`, `MAVSDK` и отдельного `MavlinkGateway`. Архитектура подтверждает возможность сбора телеметрии, контроля параметров и миссии, выявления ряда аномалий, контекстной оценки риска и журналирования событий безопасности без модификации исходного кода PX4.  
Подтверждение по исходному коду: [security_agent/app.py](../security_agent/app.py), строки 43-109; [security_agent/gateway.py](../security_agent/gateway.py), строки 82-321.

Корректно утверждать, что проект реализует программный прототип стенда, ориентированный на платформу `Holybro X500 v2 + Pixhawk 6C + PX4`, но не подтверждает аппаратную валидацию этой платформы и не доказывает детерминированность автопилота. Поэтому архитектуру следует описывать как текущую реализацию `SITL`-стенда и основу для дальнейшей адаптации к целевой аппаратной платформе, а не как завершенную аппаратно подтвержденную систему.  
Подтверждение по исходному коду: [sim_stack.json](../sim_stack.json), строки 5-13; [tools/architecture_report.py](../tools/architecture_report.py), строки 51-100, 159-175; [validation/README.md](../validation/README.md), строки 18-43.
